Code styling and structure conventions

Below is the list of code writing rules that I adopt when developing Carrot. Most of them are based on personal choice rather than a logical decision. However, it is important to have consistency, so Carrot's code will have to follow all of them. It should also be mentioned that most of them are adapted from Symfony's Coding Standards.

Naming

  • Class names are PascalCased. However, acronyms and abbreviations should remain uppercase. Example: FooBar, HTTPRoute.
  • Method names and variable names are camelCased. Like class names, acronyms and abreviations remain in their uppercase form. Example: haveSomeSleep(), getURI().
  • Interface names should be suffixed with Interface. Example: RouteInterface.

Indentation

  • Indent to four spaces, and do not use tabs. You should be able to configure your editor to convert the tab keys to four spaces.
  • Use Allman/BSD style in indenting your curly brackets. This means {} always have their own line. For example: 
    class IndentingExample
{
    public function someMethod()
    {
        echo 'Hello World!';
    }
}
  • Put space after control statements that is followed by round brackets like if, switch, while, foreach and for. Example: if ($var) instead of if($var).
  • Always use curly brackets {} after control statements even if the block consists of only one statement. This makes it easier to spot and read them.
  • Do not put spaces after opening round bracket or before closing it. Example: if ($var) instead of if ( $var ).
  • If round bracket contents are too many to write in a single line, you may put opening round bracket on the first line and the contents indented forward, the closing round bracket have its own line. Example: 
    $groceries = new GroceryList(
    'apple',
    'banana',
    'bread',
    'butter',
    'water'
);

Operators

  • Add spaces before and after all operators.
  • Use AND and OR keyword instead of && and ||.
  • Use ternary operators sparingly, favoring the if block even for simple cases.

Comments

  • Always have a doc block on top of all methods, class, and properties.
  • Protected methods must have @see tags in its doc block that acts as a list of methods that call the protected methods directly.
  • When mentioning a class from another namespace inside a comment, doc block or otherwise, write it as if you are writing the class name as a code in the class instead of a comment. This means you have to take the use keyword into account.
  • Always use // to start inline comments, not # or /* */.

Constants and booleans

  • Use uppercase to denote booleans, e.g. TRUE, FALSE.
  • Use uppercase to denote PHP native constants and null value, e.g. ENT_QUOTES, NULL.

Strings

  • Use single quotes 'string' whenever possible. However, do not hesitate to use double quotes "string {$var}" if you think it will make your code more readable.
  • When using double quotes to mix strings with variables, always wrap the variable with curly brackets. Example: "This is a {$variable}".

Class structure

  • PHP files should not end with PHP closing tag ?>. This is so that characters do not get sent accidentally to the client.
  • The usage of var is not allowed. Declare the property's visibility explicitly.
  • Declare class properties before you declare methods.
  • Public methods are declared before protected methods. Private methods are declared after protected methods.
  • If the class uses a class from another namespace that is not its subnamespace, always use the use keyword.
  • The use keyword should be declared only once, with used classes separated by commas.
